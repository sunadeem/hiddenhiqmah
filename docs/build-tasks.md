## Suggested build sequence

Ordered by dependency, not just priority. Per-feature task tables follow below.

**Phase 0 — finish the new Home** _(no new infra; mostly shipped)_
- Routing onboarding that sets `tunedFor` on first run · Daily Path live per-step state (map steps → adapter items → checkmarks → all-done state) · deep-link the Read step to the real resume position · animation/empty-state polish · confirm web parity.

**Phase 1 — the sync foundation** _(the big unlock — gates Circles, Family, cross-device Hifz)_
- **Supabase sync with conflict-resolution** (event-log/CRDT — last-write-wins corrupts irreplaceable hifz/streak data): XL. · `profiles` table (display name + avatar): M.

**Phase 2 — community + family** _(after sync)_
- **Circles** real backend: data model → CRUD/membership → invite/join → shared goal (khatmah) tracking → du'a reactions → realtime presence. · **Family** multi-profile + per-profile data scoping → parent dashboard → teacher/class roster + assignments.

**Phase 3 — Qur'an depth + retention**
- **Understanding layer** (tap-a-word): blocked on word-level data licensing — corpus.quran.com is GPL, so source from QUL or generate (see the content-licensing audit in memory) before building the reader sheet + tutor wiring. · **Hifz coach**: SRS scheduling + review flow first (works local), reciter-tolerant follow-along alignment is the XL/ML-risk piece, mistake-detection gated as Plus; cross-device needs Phase 1.

**Can slot independently**
- **Ramadan mode** — uses Hijri activation (`isRamadanActive` already exists) + composes with the Home-style system; mostly self-contained. Schedule ahead of Ramadan.
- **Humane streaks** — modifies the existing daily-adapter streak logic (mercy days, pauses, qada); no new infra, medium effort.

**Before scaling content / localization**
- **i18n re-architecture** (separate translation editions from verse data + a string layer + language-native AI): XL prereq — do *before* adding more English content or the Indonesian/Urdu push.
